Responsive Design

One of the most important concepts introduced last week was mobile-first design. This is because more and more users access the web on mobile devices. One source, Statista, states:

As of February 2017, mobile devices excluding tablets accounted for 49.73 percent of web page views worldwide. Asia and Africa had a higher than average mobile traffic share with 65.1 percent and 59.49 percent of mobile traffic respectively.

And this number is only projected to increase:

In 2020, the number of smartphone users worldwide is projected to reach 2.87 billion, up from 2.1 billion in 2016.

Because half or more of users view sites on smartphones, it's more important than ever that interfaces gracefully reconfigure themselves to fit all screen sizes.

We've already practiced media queries that do this. This week we'll explore more advanced features that allow applications to dynamically alter style and structure based on screen size. This study is called responsive design. We'll discuss things like writing code that elegantly rearranges and resizes content, special responsive considerations for images, writing responsive Sass, best practices for responsive designs, cutting-edge responsive CSS3 features, and more.

Development Environments

The second big concept we'll explore is Webpack. Webpack is a module bundler. This is a tool that collects all source code for a project, and concatenates it into one central file. This both dramatically decreases load time in the browser, and maintains an organized development environment. For these reasons, Webpack has exploded in popularity since its 2015 introduction and is now considered an industry standard for creating development environments.

Because this tool is so prevalent, we thoroughly practice it in the JavaScript course all students take. However, Webpack is in especially high demand in front end development. We've found that many (if not most) local front end job postings mention it. As such, it's important your portfolio contain examples of integrating Webpack with the front end-focused concepts of this course.

But we felt it unfair to require front end studnets complete extra work in the JavaScript course, where Weback is usually introduced, to create these work examples. This would require extra hours spend on each JavaScript project, subtracting from the already-limited time to practice the complex concepts introduced in that course.

Here's what we'll do instead: You'll receive early exposure to Webpack this week, beginning Monday evening, to craft portfolio projects that demonstrate high-demand front end skills. The trade-off? A few concepts in your first week of JavaScript will be review. Slightly more homework this week, in exchange for more review-oriented homework in three weeks, and a portfolio that will better match demands of the job market.

Based on these numbers alone, it's overwhelmingly clear web users will continue viewing content on a variety of screen sizes. This means websites must ensure their content is presentable and functional on all possible viewport sizes.

This lesson will begin exploring how developers tackle this challenge, as we discuss responsive web design, what it means, and even observe example firsthand. Then in the next few lessons we'll review how to use things like relative units, media queries, and fluid images to begin making our own sites responsive to any screen size.

Responsive Web Design

Around 2009, people began using words like fluid, elastic, and liquid to describe web layouts. Later in 2010 Ethan Marcotte officially coined the term responsive web design in his article Responsive Web Design. He stated:

Recently, an emergent discipline called “responsive architecture” has begun asking how physical spaces can respond to the presence of people passing through them...In their book Interactive Architecture, Michael Fox and Miles Kemp described this more adaptive approach as “a multiple-loop system in which one enters into a conversation; a continual and constructive information exchange.” I think that’s a subtle yet powerful distinction: rather than creating immutable, unchanging spaces that define a particular experience, they suggest inhabitant and structure can—and should—mutually influence each other.

This is our way forward. Rather than tailoring disconnected designs to each of an ever-increasing number of web devices, we can treat them as facets of the same experience. We can design for an optimal viewing experience, but embed standards-based technologies into our designs to make them not only more flexible, but more adaptive to the media that renders them. In short, we need to practice responsive web design. But how?

While Marcotte's original statement is dense, even challenging to understand upon first reading, know that its core idea is simple: User interfaces should flexibly adapt based on screen size, to offer users similar experiences regardless of their device.

In the Value Units prework lesson we discussed the term relative units, explaining:

Relative units...are relative to things like the parent container, an element's font size, or the size of the browser window. (For instance, if we specify an image should be 50% the width of its parent container, it will differ in size depending on the size of that parent!)

We learned that em, rem, vw, and % (percentage) values are relative because they flexibly change in relation to other elements (usually their parent). For example, if we declare a div should be 50% as wide as its parent, it doesn't matter if that parent is 1000px wide on a desktop, or 200px on a smartphone; the child div will always be half as wide.

This dynamic flexibility makes relative units a crucial part of responsive design, because they automatically ensure elements maintain a size in consistent proportion with other elements, regardless of viewport size. This save us from writing tons of media queries to resize each individual item. In this lesson we'll discuss how to best use relative units to create responsive layouts.

(If the concept relative units feels unfamiliar, review the Value Units lesson before continuing.)

Relative Units

Many relative units used in responsive design (besides viewport units) actually originate from the very first version of CSS! But their use in responsive design is considerably more far-reaching. As usage of relative units grows increasingly broad, best practices become more involved.

Here are the most important best practices for relative CSS units in responsive design:

Use px Units for Constant Sizing

Always use px units for elements that should stay the same dimensions regardless of viewport size. That is, if we don't want an item to resize with the viewport.

Elements we want to stay consistent, regardless of viewport size, include things like the width of a document's first parent container (often denoted with the class container), borders and padding, or the width of a line break.

The CSS below, for instance, follows this rule:

html {
          box-sizing: border-box;
          font-size: 100%;
        }

        *, *:before, *:after {
          box-sizing: inherit;
        }

        body {
          margin: 0;
          padding: 0;
        }

        .container {
          border-left: 2px solid lightSeaGreen;
          border-right: 2px solid lightSeaGreen;
          margin: 0 auto;
          max-width: 1400px;
          padding: 15px;
        }

        .main-content {
          display: flex;
          flex-direction: column;
        }

        .content-box {
          width: 48%;
          justify-content: space-between;
        }

  • px creates rigid content values, like the max-width and padding of the parent .container. The maximum is just that...the maximum, we don't want this upper limit to change in size.

  • Styles for children of this parent .container use relative values; primarily percentages. This ensures their size maintains a consistent, proportional relationship with the parent's size.

Avoid px in Child Elements

In child elements, use em, rem, percentages, or viewport units (vw) whenever possible.

Using absolute units like px on child elements can lead to pretty random numbers. These are called floating or magic numbers because they're difficult to understand almost immediately after they're written. As the Magic Numbers in CSS article by Chris Coyier describes:

Despite the super fun sounding name, magic numbers are a bad thing...[they are] just some number plunked into the code that is probably vital to things working correctly but are very difficult for anyone not intimately familiar with the code to understand...

  1. A flexible, grid-based layout
  2. Flexible images and media

Media Queries

As discussed earlier in the course, a media query is a block of CSS applied only when specific conditions about the user's viewport are true. This allows us to change layout structure and style based on viewport width. The point at which a condition becomes true, and the query's CSS is applied, is called the breakpoint.

Determining Breakpoints

Over time, phones, tablets and other devices have created widely-recognized pixel ranges often used in media queries. For example, as outlined here, most tablets are between 600 and 900 pixels in portrait orientation. So you may assume that breakpoints should correspond exactly with these common device sizes.

But that's actually a dangerously incorrect assumption! Determining where breakpoints should lie differs greatly between applications, and requires more careful thought and planning than referring to common screen sizes. Let's discuss best practices for effective breakpoints in responsive designs.

Required Reading: The 100% Correct Way to do CSS Breakpoints

First off, read David Gilbertson's tips for constructing breakpoints in his article, The 100% Correct Way to do CSS Breakpoints.

Breakpoint Best Practices

As Gilbertson explains, configuring breakpoints is really tricky. Fortunately, there are rules we can follow to help:

Content First

Breakpoints should be determined based on the specific application's content, not an outside standard like common smartphone sizes. The breakpoints in Gilbertson's the article are a great guideline, but your breakpoints for your applications should be determined based on the needs of your unique content.

For example, if there's a high-resolution image we want featured prominently on a page, but a 'standard-issue' breakpoint cuts it off or sizes it in an unflattering manner, that's a big indication we should make the breakpoint work for the content, instead of forcing the content into the breakpoint.

min- or max-width for Mobile or Desktop

Last week we learned how to follow mobile-first design, sketching and wireframing mobile layouts first, before desktop versions. But as long as we finalize mobile designs first, it doesn't actually matter which we code first. (But your future employer or internship company may have their own workflow preferences.)

And, conveniently enough, the version we opt to code first will dictate how we set breakpoints! Here are helpful rules to reference:

  • If you begin coding the mobile layout first, write breakpoints using min-width, because you'll be moving from smaller mobile screens up toward larger desktop screens.

  • If you start with desktop, write them with max-width, because you'll move from a larger desktop viewport into a smaller mobile screen.

(If min-width and max-width don't ring a bell, review rules for these expressions in the Viewports and Media Queries lesson.)

Other Considerations

But these are just the basics of responsive breakpoints and media queries. You'll see lots of tutorials online suggesting different approaches. We think the recommendations here are easiest and most efficient for beginners, but as you grow more comfortable with responsive design, feel free to try different approaches!

As we've already discussed, responsive web layouts are composed of three primary ingredients:

  1. A flexible, grid-based layout
  2. Flexible images and media
  3. Media queries

The last lesson discussed media queries. But what about that first item on this list? In this lesson we'll discuss what a flexible grid is, what they do, and how they helps create the dynamically-responsive layouts users have grown to expect.

Flexible (or Fluid) Grids

The term flexible grid was also introduced by Ethan Marcotte (yep, same person who coined responsive design) in an article titled Fluid Grids. The article tells the story of redesigning a client's content-heavy site. The client wanted their design to resize with the browser window, but this created a serious hurdle for Marcotte...you see, there simply weren't any grid-based responsive options in 2009. They were all very rigid. So he had to develop his own.

After a series of attempts he found a solution:

...as I was working on relative font sizing, that’s when it hit me: every aspect of the grid--and the elements laid upon it--can be expressed as a proportion relative to its container...This will allow us to convert our design’s pixel-based widths into percentages, and keep the proportions of our grid intact as it resizes.

In short, we’ll have a fluid grid.

This is an important distinction. Fluid/flexible grids do not inherently refer to flexboxes, or even a feature called CSS grids (we'll explore what CSS grids are in the next lesson). They're a general type of layout that has existed long before either of these features. We could create a flexible grid using CSS grid or flexbox features, sure, but they're not the only way.

(Essentially, just keep in mind "flexible grid" is a general term, whereas CSS Grid and Flexbox are specific features in CSS.)

Notice how the different elements and areas of the layout line up perfectly on this grid. Then notice how each area of the grid retains proportions as the viewport resizes:

As we can see, fluid grids are the cumulative use of relative values and a media query-based grid system. The goal with any grid layout is to be structural rigid enough to organize content cleanly and consistently, yet still flexible enough to dynamically resize for any viewport.

Grids are a construct that abstract other tools. So it's admittedly difficult to write rules for how they should be used. There's no single best practice for flexible grid-based layouts. It's dependent on needs of the specific project. That said, the priorities of flexible grids lie on a spectrum;

  • On one end exists the snappy, rigid design focusing on displaying content in the clearest, most consistent way possible.

  • On the other end exist flowing grids that maintain the same viewport ratio regardless of screen size. These grids gently decrease or increase their content sizes as the browser window expands or contracts, as opposed to "snapping".

And responsive layouts can exist anywhere between these two ends of the spectrum.

Examples

Here are two examples that prioritize these goals differently. Visit each and expand/contract your browser window to observe how they change:

Deutsche & Japaner

This slick agency site has an highly-stylized grid system brimming with media. As we resize the browser window, we see grid sections are dictated by relative values (pictures, for example, take up 100%, 50%, or 25% of the screen width). At media breakpoints the grid changes shape drastically.

NPR News

In contrast, NPR's homepage uses a simple grid with lots of margin to keep content cleanly separated. Notice there are also fewer breakpoints. This prevents article titles from "snapping" or "jumping" out of visual focus, as they would if the grid changed drastically over more than three breakpoints.

What other differences do you notice between these two? Is there one you're more drawn toward? Do you think different content is best represented by different approaches? How will the grids you use maximize user experience?

This is the question inherent in responsive design.

Grid Anatomy and Terminology

Let's first clarify CSS grid anatomy and terminology. Some of this will be review from the video in the previous lesson.

Consider the following diagram from gitconnected's article How to Get a Quick Start with CSS Grid Layout:

Let's walk through each part depicted here:

Grid Container

A grid container is the element that contains the grid itself. We create it by applying a display: grid; style to it, like this:

.example-grid {
    display: grid;
  }

As soon as an element is a grid container, all its children automatically become grid items, and span the full width of the parent grid container.

Grid Items

Grid items are simply the items organized by the grid. They must be direct children of the grid container. All direct children of grid containers are automatically grid items.

Grid Lines

Grid lines are the intersecting lines that make up the grid itself.

  • There are column lines that run top-to-bottom;

  • and row lines that run right-to-left, as seen in the image above.

Grid Cell

A grid cell is an individual "box" in the grid (one of the nine squares composing the grid seen above).

Grid Track

A grid track is the space between two adjacent grid lines. Each entire row or column in the grid is considered a grid track.

Grid Area

Grid area refers to the combined area of multiple cells (but it must be a rectangle, and cannot be an L-shape. See diagram above.)

Grid Gap

Grid gap refers to optional space between each column and row, as emphasized in the image below (it's not included in the diagram above). Sometimes specifically referred to as column gap, or row gap.

Using this Lesson

Each of the following headers links to the stylesheet used in its example. As you read through each example's benefits and drawbacks, visit the stylesheet and observe what styles are used, and how the three differ.

1. Fluid Layout with Relative Units

This example presents responsive content without media queries or grids. Instead, we're only using relative units to transform content to fit different viewport sizes.

  • As discussed in the Relative Units lesson, padding, border, and container values are set with px, but other values are percentages of either a parent element (using relative percentage % values) or the viewport (using viewport units vw).

  • Notice the font-size for the entire document is a function of vw. This makes the text size responsive.

  • Also, as in more traditional box model styling, the dimensions of the parent content elements (.header, .content-big, and .content-small) all derive their dimensions from the content itself, instead of being set explicitly.

This can be an effective way to style a page with static text that doesn't change. But much of what makes this layout fluid will also make it difficult to retain its shape.

For example, if content ever changes, or the layout must shift in small ways, writing explicit rules into this stylesheet will cause challenges. What if, for example, we want .content-big to be a perfect square in all orientations?

2. Multiple Layouts with Media Queries

We learned it's best practice to use relative units like em, rem, and % for text and page content as much as possible. This next stylesheet intentionally breaks that rule to demonstrate what a media query achieves for us. It uses only media queries, and no relative values.

  • Notice the different layouts we see between each breakpoint are entirely static. They only "snap" from one layout to the next when breakpoints are applied, and do not gently expand or shrink with the viewport like the previous example.

  • The only content that shifts with the window size is the header, because its display is block by default, and automatically takes up 100% of its parent width.

There's nothing seriously wrong with this approach either. But part of modern web design is creating layouts that work well for text and media content. What would happen if this project included a banner image before the main content?

3. Relative Units, Media Queries, and Flexbox

This third approach uses relative units, media queries, and flexbox together. This allows us to exploit both fluid content and grid systems, giving us the best combination of readable content and fluid transitions between layouts.

  • Although this page doesn't include images, the structure of its stylesheet would be easy to adapt to include fluid images, which we'll discuss further in the next lesson.

  • Also, by using the media queries and grids, it's easier for us to follow best practices of "Outside In".

Goal: Focus on crafting cohesive web layouts that dynamically resize and reorganize content for any viewport size, paying special attention to the new CSS grid concepts.

Imagine we're front end developers at a large company that creates and maintains huge apps and websites. Each has hundreds of pages, tons of Sass styles, and countless interfaces to be designed and maintained.

Now imagine these applications used only the HTML, CSS/Sass and JavaScript techniques we've covered so far. Their index.html pages could have hundreds of script tags, each loading a necessary file or tool (like Bootstrap, jQuery, and more). This would be messy. It would also be slow. You see, each of those files takes time to load. Time that adds up quickly. And if a site takes too long to load, many users just leave, losing the company customers and revenue.

This is a real-life struggle for many companies. In fact, Amazon estimated a single second of lag time would cost them a staggering $1.6 billion in yearly sales. Similarly, Google has determined their traffic drops by 20% if search results take an extra half second to load! In a world where technology responds increasingly fast, users just don't wait. So it's up to us, the developers, to ensure speedy load times.

One common way to decrease load time is to put all code (both code written by developers, and code for outside tools like jQuery or Bootstrap) into a single file. This is called bundling or module bundling. It's quicker because the browser only has to load one file. This is faster than loading hundreds of smaller files linked in <script> tags, even if the single file is pretty hefty.

Bundling code like this is really common nowadays. So we're going to practice it too. But we must first set up a development environment to handle this process. Let's learn about them now.

Modern JavaScript Development

We'll learn what they do, why developers use them, and associated vocabulary. Then in the next few lessons we'll walk through creating our own development environment together.

Dependencies

Do you remember importing Bootstrap into projects in Intro? Or downloading and linking a jQuery file? These outside tools contained code we didn't write. But we weren't plagiarizing! This code is free and available for anyone to use.

Pieces of outside code we bring into our projects like this are called dependencies. There are outside libraries (chunks of code designed around a specific purpose that we can bring into a project as a dependency) for almost anything! For instance, a package called Moment helps gracefully work with dates and times, libraries like documentation.js help auto-generate documentation for big projects, and dependencies like Animate.css even create CSS animations. (But don't worry about these right now, they're just examples.)

Cool thing is, anyone can publish a library for others to use as a dependency! Whether a big dev team at a well-known company (like the React library created by Facebook), or a single developer working on a tool at home.

Package Managers

We practiced locating, downloading, and linking files for Bootstrap and jQuery into our projects in Intro to Programming. But there's actually a much easier, quicker, and more efficient way to handle this. Something called a package manager allows us to download and install dependencies directly through the command line!

Package managers also automatically create and maintain a file called a manifest that lists all installed dependencies. It automatically updates this list as we add or remove dependencies, so we always know what's installed. The manifest ensures other developers who clone our project download and install the correct dependencies in their copy too.

Module Bundler

A module bundler collects all project source code and concatenates it into a big central file. Using a module bundler has recently become best practice for several reasons:

  • Decreased load time. Projects often use many tools. Each must be imported into the project and/or loaded with <script> tags. The browser loads them individually, which takes time. But combining code from all dependencies into one file (known as a bundle) decreases this load time dramatically, because the browser must only load a single file.

  • Minified code. Most bundlers also minify code. Minification is the process of removing unnecessary characters (whitespace, comments, etc.); things that humans need to easily read code, but computers do not. Removing these characters makes our file smaller. Smaller files load faster. If you've ever seen a file (like a Bootstrap stylesheet) with .min in the name you've encountered minified code.

That $ npm install command prompts npm to download and install the specific versions of all dependencies listed in the project.json manifest. And this ability to install everything with a single command is actually so quick and painless that it allows us to omit dependencies from repos entirely.

Omitting Dependency Code

In fact, it's considered best practice not to commit outside dependency code (that is, anything in the node_modules directory) to Git and GitHub repos. This ensures repos contain only (or at least mostly) our own code, and keeps them clean, organized, and easy to navigate.

When other developers (or ourselves in the future) want to retrieve and run the project they can clone down the repo and simply run $ npm install to snag all dependencies.

This means we must make sure our package.json manifest is always committed, and our node_modules directory is always ignored. We can handle this with a simple .gitignore file.

.gitignore Review

Back in Intro the Git Configurations lesson showed us how to omit certain files from all repos on a global level. But we can also add .gitignore files to individual projects.

We'll create our own .gitignore in the top-level of our practice project now, and add the following:

.gitignore
node_modules/
    .DS_Store

A few things to keep in mind:

  • .gitignore files always reside in the top-level of a project directory.

  • The .DS_Store we listed inside our .gitignore, seen above, is an unnecessary Mac-specific file that doesn't need to be committed.

  • node-modules/ refers to the directory containing source code for dependencies installed with npm.

  • The .gitignore file must be added and committed before we commit node_modules. (But accidentally committing node_modules before .gitignore is a common mistake. There's an easy solution explained below.)

Common .gitignore Mistake

If you Git is tracking a file that should actually be ignored, make sure to add that file to .gitignore. Then run $ git rm --cached [FILE-OR-DIRECTORY-NAME], where [FILE-OR-DIRECTORY-NAME] is the name of the file or directory that shouldn't be tracked.

So if we ever accidentally commit node_modules before adding and committing our .gitignore file specifying to ignore this directory, we can run $ git rm --cached node_modules to retroactively remove it from the repo, and begin ignoring it instead.

From now on, your projects should always include a .gitignore file with node_modules listed.

Uninstalling Dependencies with npm

Continuing with our practice project, we won't actually use Bootstrap or jQuery this week. We just used these familiar dependencies to practice npm.

Let's uninstall them from our project now. Npm has another easy command for that too:

$ npm uninstall bootstrap
    $ npm uninstall jquery

Just like that, we should see the packages disappear from our node_modules directory and package.json manifest.

We've learned that modern development environments include dependencies, a package manager, and a module bundler. In the last few lessons we discussed package managers and dependencies, and practiced using them in a sample project.

Next let's explore module bundlers. Specifically Webpack, the industry bundler of choice.

Introduction to Webpack

As you'll recall from a few lessons ago:

A module bundler collects all project source code and concatenates it into a big central file. Using a module bundler has recently become best practice for several reasons:

  • Decreased load time. Projects often use many tools. Each must be imported into the project and/or loaded with <script> tags. The browser loads them individually, which takes time. But combining code from all dependencies into one file (known as a bundle) decreases this load time dramatically, because the browser must only load a single file.

  • Minified code. Most bundlers also minify code. Minification is the process of removing unnecessary characters (whitespace, comments, etc.); things that humans need to easily read code, but computers do not. Removing these characters makes our file smaller. Smaller files load faster. If you've ever seen a file (like a Bootstrap stylesheet) with .min in the name you've encountered minified code.

We'll use Webpack as our module bundler. In addition to general benefits listed above, Webpack offers a few of its own helpful features:

  • Flexible Configurations: Webpack supports things called plugins and loaders that configure it to work with almost any other technology or language.

  • Development Server: It also offers a development server that can be configured to re-bundle code every time it changes, and serve it up in the browser to view. (We'll work on this in an upcoming lesson.)

Installing Webpack

We'll use npm to install Webpack into the practice project we've been working with. Run the following command in your project directory. (It's different than commands from previous lessons. We'll discuss why in a moment.)

$ npm install [email protected] --save-dev

There are two new parts to this command:

  1. The @4.0.1 appended to the end of the package name. This is a specific package or dependency version.

  2. The --save-dev bit added to the very end. This specifies that Webpack is a development dependencies.

Let's discuss each of these new additions:

1. Package Versions

Whenever we want npm to install a specific version of a dependency we add the version number to the installation command. So by stating $ npm install [email protected] we're instructing npm to specifically download and install Webpack version 4.0.1.

If a version number is not specified (as seen in the previous lesson in commands like $ npm install bootstrap) npm defaults to the latest stable version.

When following Epicodus curriculum, always use the exact installation commands provided in lessons, including exact version numbers. Not all packages play nicely with every version of all other packages. And you don't want to detangle complex incompatible package version issues while trying to practice basics, or learn new tools.

Carefully curating versions like this isn't specific to Epicodus. It's part of professional web development. Again, not every dependency works perfectly with every other dependency. So when we find the perfect combination, we keep those exact versions in our package.json manifest. This is called version pinning. Development teams then continue using those pinned versions until they absolutely need a feature a newer version offers, until the pinned version is retired, or until the pinned version no longer functions correctly with new code added to the project.

On a big professional project, upgrading dependency versions is really tricky! It's an all-hands-on-deck project the entire development team sets time aside time for. It requires tons of testing and research. Bringing in a new version also increases the chance for bugs. As such, professional development teams don't upgrade dependencies just because newer versions were released. Have you ever heard the classic saying "If it ain't broke, don't fix it"? It's best practice to follow this same adage when dealing with versioning too.

This in mind, it's common to see even the most well-maintained codebases using dependencies multiple versions behind the latest releases. This is normal. Always use the exact versions depicted in our curriculum, even if it's not the absolute most recent version.

Essentially: Newer does not mean better. Older does not mean outdated. And if it isn't broken, developers don't risk bugs and outages to update to latest versions just because they're newer. So you shouldn't either.

2. Development Dependencies

Moving on, that --save-dev bit on the end of the $ npm install [email protected] --save-dev command above is new too.

Open up the practice project's package.json manifest file and take another peek:

package.json
{
  "name": "modern-environment-practice",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC",
  "dependencies": {},
  "devDependencies": {
    "webpack": "^4.0.1"
  }
}

Notice there's a new section called devDependencies. This stands for development dependencies, which refers to dependencies only used while the project is in development. This is different than "normal" dependencies (in the dependencies section of package.json) that will be used both while the project is in development, and when the final, published product is run by users in their browsers.

Remember jQuery? When sites we developed in Intro used jQuery they required that functionality in both production and development versions. If we were using a jQuery method to add a hover effect to a button, for instance, that jQuery method had to be available both when we're developing (so we, the developers, can test it out) and also in the final published version (so the users see that nifty hover effect too.) This is an example of a "normal" dependency.

But Webpack is a module bundler. It smushes code into a central bundle file. Once that code is published online, Webpack doesn't need to do anything to our code until the next time we, the developers, update and re-publish it. It will only ever be used during development, so it's considered a development dependency.

Don't worry if the difference between dependency types isn't immediately clear. We'll continue to denote which dependencies to install as which type. You'll have the rest of the program to solidify this knowledge.

Webpack CLI

Moving on, Webpack is now added to our project as a development dependency. When we use Webpack we'll run a command in our terminal to tell it to bundle up our code. (Similar to how we run a command to compile Sass.)

To use Webpack in the terminal we need another package called Webpack CLI. This package grants us the ability to call Webpack from the command line (CLI stands for "command line interface"). We can install it now:

$ npm install [email protected] --save-dev

With this installed, we can now run the following in our project directory:

$ webpack

This tells Webpack to bundle our project's code. But hey! We actually get an error:

Hash: 86cb31cfb7da0752b199
Version: webpack 4.0.1
Time: 67ms
Built at:  

WARNING in configuration
The 'mode' option has not been set. Set 'mode' option to 'development' or 'production' to enable defaults for this environment.

ERROR in Entry module not found: Error: Can't resolve './src' in '/Users/epicodus_student/Desktop/modern-environment-practice'

Here we have both a warning message, and an error message.

The warning states we need to tell Webpack whether this is a production or development environment. We do that by adding a flag to our command, like this:

$ webpack --mode development

We'll add this flag to all commands instructing Webpack to bundle code.

The warning says it cannot find a ./src directory in our project. By default, Webpack 4.0+ looks for an entry point file at src/index.js, as this is common convention.

An entry point is the file that instructs the module bundler how to build and bundle the application. It's a bit like an instruction manual. Let's construct one in our practice project now!

Entry Point File

We'll create an src directory at the top level of our project. src stands for "source", or "source code". Source code is the custom code we write--not code imported from dependencies, or code in the bundle file Webpack produces. This means all code we write resides in src.

Within our new src directory, we'll add an index.js file. For now we'll just throw a basic console.log() in it:

src/index.js
console.log("hey!")

Now let's run Webpack again with the appropriate command:

$ webpack --mode development

And we should see something like this appear in the console:

Hash: 6591330ed601b5a8a96b
Version: webpack 4.0.1
Time: 82ms
Built at:  
  Asset      Size  Chunks             Chunk Names
main.js  2.84 KiB    main  [emitted]  main
Entrypoint main = main.js
[./src/index.js] 27 bytes {main} [built]

This means Webpack has successfully bundled our code! Check out our project directory:

modern-environment-practice
├── dist
│   └── main.js
├── node_modules
│   └── ((content omitted for brevity))
├── package-lock.json
├── package.json
├── src
│   └── index.js
├── webpack.config.js
└── .gitignore

That dist directory is new! It contains a single file, main.js which is the code Webpack has bundled up for us! "dist" stands for distribution, and is common naming convention for directories containing bundled code, because this code will be distributed to the browser.

It's important that we never, ever edit code in dist directly. This code is only ever altered by Webpack, not us. We edit our project's code in the src directory, then rely upon Webpack to turn our code from src into the bundle distributed in the browser in dist.

Never edit dist. Only edit src.

Ignore Dist

Remember how we added our node_modules directory to our .gitignore file in the last lesson to omit it from repos? We did this because other developers can simply run $ npm install to download all dependency code in node_modules, so we don't need to save them in our repo.

Well, we don't need to include this dist directory in our repo either, for a very similar reason. Other developers (or ourselves in the future) can simply run $ webpack --mode development to bundle code into the main.js file in dist. So we don't need to save it in our repository. We'll add this directory to our .gitignore file now:

.gitignore
node_modules/
.DS_Store
dist/

Make sure to do this in all future projects that use Webpack, too!. And remember, if we ever accidentally commit dist to our repo before adding and committing a .gitignore file ignoring this directory, we can run $ git rm --cached dist to retroactively remove it from the repo, and begin ignoring it instead.

Great! We now know how to install Webpack in our projects, the commands to run to instruct it to bundle up our code, and best practice for ignoring those bundles. In the last lesson we attempted to run the $ webpack command in our project directory, but received a warning telling us the command must specify whether the project is in development, or production mode. So we added this info to our command, like this...

$ webpack --mode development

...And it worked without error!

Additional information added to a command, preceded by hyphens like this, are called flags.

npm Scripts

Remembering to add the --mode development flag to our Webpack command isn't too inconvenient. But we can actually make this process easier.

Notice this section in package.json?

package.json
...
    "scripts": {
      "test": "echo \"Error: no test specified\" && exit 1"
    }
  ...

scripts refers to code that runs in the terminal to complete a task. And test is a type of script that came with this file by default.

Running npm Scripts

We run scripts included in package.json by preceding them with npm run in the command line. So we can test out this test script by running the following in the project directory:

$ npm run test

It returns this:

echo "Error: no test specified" && exit 1

  Error: no test specified
  npm ERR! Test failed.  See above for more details.

Notice this matches text from package.json:

package.json
...
    "scripts": {
      "test": "echo \"Error: no test specified\" && exit 1"
    }
  ...

So. We ran the $ npm run test script, and npm spat the content associated with test in package.json into the terminal. Following this same pattern, we can actually define scripts that insert whatever we want into terminal; including other commands!

Writing npm Scripts

We'll add the following to package.json:

package.json
...
  "scripts": {
    "build": "webpack --mode development",
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  ...

Now if we run $ npm run build, we should see Webpack runs the $ webpack --mode development for us!

It might not feel efficient to just map one command to another command like this, but it's actually best practice! You see, build is standard naming convention for scripts that compile or bundle a project. So by mapping whatever our bundling command is to this standardized $ npm run build, any developer can use the same industry standard to launch our project. And we don't have to remember that --mode development flag either!

Remember to map your Webpack bundling command to the npm build command in all future projects. We'll explore other helpful npm scripts in future lessons too!
